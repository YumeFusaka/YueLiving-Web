# 悦居管家三端平台设计文档

## 1. 目标

在保留现有技术框架不变的前提下，完成单小区物业管理平台的三端业务落地：

- 业主端：注册登录、我的房产、费用中心、在线报修、公告通知、个人中心
- 物业管理端：业主管理、房产资源管理、费用管理、报修工单管理、公告管理、数据统计、个人中心
- 系统管理员端：管理员管理、系统配置管理、数据总览

本次实现允许优化和重构现有业务代码，但尽量不改动基础框架代码，如 `axios` 封装、JWT 工具、基础拦截器、前端项目脚手架。

## 2. 现状判断

当前项目已具备以下基础：

- 前端为 `Vue 3 + Vite + Element Plus + Pinia + Vue Router`
- 后端为 `Spring Boot + MyBatis-Plus + JWT`
- 已有基础表结构：用户、角色、房产、账单、报修、公告
- 已有基础页面和 API 雏形

当前主要问题：

- 角色页面混用严重，缺少明确的三端边界
- 账单、报修、房产等实体缺少支撑完整业务流程的字段
- 后端接口以基础 CRUD 为主，缺少面向业务场景的查询与统计接口
- 缺少系统配置、操作日志、账单生成、报修分类、统计总览等模块
- 部分权限与数据归属校验存在错误或不足

## 3. 范围与边界

### 3.1 范围

本次实现只支持单小区，不引入多小区、多项目、多租户概念。

### 3.2 角色边界

- 业主：只能操作与自己绑定的房产、账单、报修、个人资料
- 物业管理员：负责小区运营数据维护与处理，不管理系统级配置
- 系统管理员：负责物业管理员账号、系统配置、平台总览，不直接承担日常物业工单处理

### 3.3 不做事项

- 不接入第三方支付，仅做“线下已缴/模拟缴费”状态流转
- 不实现短信验证码登录，注册登录仍基于账号密码；手机号作为注册信息和可检索字段
- 不引入对象存储，图片继续走本地上传目录
- 不引入新的大型前端状态框架或后端安全框架重写

## 4. 总体方案

### 4.1 后端

保留现有 Spring Boot、MyBatis-Plus、JWT 与权限注解模型，在此基础上重构业务接口层和数据模型。

后端按业务域组织：

- 用户与认证
- 房产资源
- 费用账单
- 报修工单
- 公告通知
- 系统配置
- 数据统计
- 操作日志

实现策略：

- 保留已有实体与控制器主线
- 对现有表做增量字段补充
- 新增少量支撑表完成配置、日志、分类、账单规则能力
- 新增更贴近页面场景的聚合查询接口，而不是只依赖基础 CRUD

### 4.2 前端

保留现有项目结构与路由体系，重构业务页面的组织方式：

- 登录后按角色展示不同导航
- 业主端、物业端、系统管理员端共用一个主布局
- 各业务页面拆成更明确的视图职责，不再把所有角色逻辑堆在同一张表内
- 公共个人中心保留统一入口，但内部展示内容按角色分组

### 4.3 设计原则

- 先保证业务闭环完整，再考虑局部样式优化
- 页面接口按“用户任务”设计，不按数据库表硬映射设计
- 在现有代码基础上演进，不做无关重构
- 所有关键状态流转必须有权限校验和归属校验

## 5. 数据模型设计

## 5.1 保留并扩展现有表

### `user`

保留现有字段，并明确使用约束：

- `role_id`：1 业主，2 物业管理员，3 系统管理员
- `status`：0 禁用，1 启用
- `phone`：允许用于注册和检索
- `real_name`：前后台展示名称

新增字段：

- `last_login_time`：最近登录时间
- `remark`：管理员备注

### `property`

保留原有楼栋、单元、房号、面积、业主绑定关系。

新增字段：

- `property_type`：住宅/车位/商铺，默认住宅
- `owner_name_snapshot`：绑定时的业主姓名快照，便于查询展示
- `bind_time`：绑定时间
- `status`：调整为枚举值，至少支持空置、已入住、装修中

### `bill`

保留原有账单主体信息。

新增字段：

- `owner_id`：账单所属业主，避免仅靠房产反查
- `bill_item_name`：账单项目名，如“2026年4月物业费”
- `unit_price`：单价
- `usage_amount`：用量，适用于水电费
- `generate_type`：自动生成/手动生成
- `remark`：费用说明
- `paid_amount`：已缴金额

状态定义：

- 0 未缴
- 1 已缴
- 2 逾期
- 3 已作废

### `repair_order`

保留原有报修主信息。

新增字段：

- `repair_type_id`：报修分类
- `contact_name`
- `contact_phone`
- `priority_level`
- `processing_result`
- `cancel_reason`
- `satisfaction_label`

状态流转定义：

- 0 待受理
- 1 已受理
- 2 处理中
- 3 已完成
- 4 已取消

### `announcement`

保留公告基础字段。

新增字段：

- `category_code`：公告分类
- `status`：草稿/已发布
- `summary`：摘要

## 5.2 新增表

### `system_config`

用于系统可维护配置，字段建议：

- `id`
- `config_key`
- `config_name`
- `config_value`
- `config_type`
- `remark`
- `create_time`
- `update_time`

配置项示例：

- 物业费默认单价
- 报修类型列表
- 公告分类列表
- 欠费逾期天数

### `operation_log`

用于记录关键操作，字段建议：

- `id`
- `operator_id`
- `operator_role_id`
- `module_name`
- `action_name`
- `target_type`
- `target_id`
- `content`
- `create_time`

### `repair_type`

报修分类独立表，字段建议：

- `id`
- `type_name`
- `sort_no`
- `status`

### `billing_rule`

用于物业费配置与自动生成规则，字段建议：

- `id`
- `bill_type`
- `unit_price`
- `cycle_type`
- `status`
- `remark`

## 6. 后端接口设计

## 6.1 用户与认证

保留现有登录注册接口，补充或调整：

- 支持用户名或手机号登录
- 登录时校验账号状态
- 注册默认创建业主账号
- 个人资料修改和密码修改继续保留
- 系统管理员可管理物业管理员账号
- 物业管理员可查看业主账号列表并启停业主账号

新增场景接口：

- 业主列表分页查询
- 管理员列表分页查询
- 启用/禁用用户
- 重置管理员密码
- 获取当前登录用户工作台信息
- 获取个人操作日志

## 6.2 房产资源

业主：

- 查看自己绑定的房产列表及详情

物业管理员：

- 房产分页查询
- 新增/编辑/删除房产
- 绑定业主
- 解绑业主
- 按楼栋/单元/房号筛选

## 6.3 费用账单

业主：

- 查看账单列表
- 按月份、状态、类型筛选
- 查看历史缴费记录
- 查看欠费提醒

物业管理员：

- 维护收费规则
- 生成账单
- 手动新增账单
- 调整异常费用
- 标记缴费状态
- 查看缴费统计报表

系统规则：

- 可按周期生成物业费账单
- 水电费支持手动录入用量和金额
- 欠费账单自动识别逾期状态

## 6.4 报修工单

业主：

- 提交报修
- 上传图片
- 查看工单进度
- 查看维修人员信息
- 完成后评价

物业管理员：

- 工单筛选查询
- 受理工单
- 指派处理人
- 更新处理状态
- 填写处理结果
- 查看效率统计

## 6.5 公告通知

业主：

- 查看公告列表
- 查看置顶公告

物业管理员：

- 草稿保存
- 发布公告
- 编辑公告
- 删除公告
- 置顶公告

## 6.6 系统配置

系统管理员：

- 维护物业费规则
- 维护报修分类
- 维护公告分类
- 维护逾期提醒参数

## 6.7 数据统计

物业管理员：

- 月度缴费率
- 未缴账单统计
- 报修状态分布
- 报修分类占比
- 满意度趋势

系统管理员：

- 全系统用户构成
- 整体缴费情况
- 报修趋势
- 平台使用情况

## 7. 前端页面设计

## 7.1 登录与注册

- 登录页保留现有入口，支持账号或手机号登录
- 注册页默认注册业主账号
- 登录成功后根据角色进入统一首页

## 7.2 布局与导航

统一主布局保留，但菜单按角色生成：

- 业主：仪表板、我的房产、费用中心、在线报修、公告通知、个人中心
- 物业管理员：仪表板、业主管理、房产资源、费用管理、报修工单、公告管理、数据统计、个人中心
- 系统管理员：仪表板、管理员管理、系统配置、数据总览、个人中心

## 7.3 业主端页面

### 仪表板

展示：

- 房产数量
- 未缴账单数
- 待处理报修数
- 最新公告

### 我的房产

表格或卡片展示房产详情，支持查看绑定状态和基础信息。

### 费用中心

分为：

- 当前待缴账单
- 历史缴费记录
- 欠费提醒

### 在线报修

分为：

- 报修提交表单
- 我的报修记录
- 工单详情与评价

### 公告通知

按置顶和普通公告分区展示。

### 个人中心

分为：

- 个人资料
- 修改密码
- 我的报修记录
- 缴费历史

## 7.4 物业管理员页面

### 仪表板

展示运营关键指标卡片和趋势统计。

### 业主管理

展示业主账号列表、状态切换、信息查询。

### 房产资源管理

支持房产 CRUD、绑定解绑业主、状态维护。

### 费用管理

支持账单查询、账单生成、费用调整、缴费状态更新、收费规则维护。

### 报修工单管理

支持工单筛选、受理、分配、状态推进、完成登记。

### 公告管理

支持新增、编辑、发布、置顶、删除。

### 数据统计

以卡片和简易图表展示统计结果。若不引入图表库，则优先使用 Element Plus 表格和进度条实现基础可视化。

### 个人中心

展示个人信息、待处理工单、已处理工单、个人操作记录。

## 7.5 系统管理员页面

### 仪表板

展示整体系统数据概览。

### 管理员管理

支持物业管理员账号新增、编辑、启停、重置密码。

### 系统配置

支持配置项列表维护、报修分类维护、公告分类维护。

### 数据总览

展示系统级统计数据。

### 个人中心

支持个人资料与密码维护，并查看个人操作记录。

## 8. 权限与校验设计

### 8.1 权限原则

- 路由层只负责基础登录校验和角色拦截
- 接口层必须再次做角色校验
- 涉及资源归属的接口必须做数据归属校验

### 8.2 关键归属校验

- 业主只能看自己的房产
- 业主只能看自己名下房产账单
- 业主只能评价自己的已完成报修
- 物业管理员不能操作系统管理员账号
- 系统管理员不直接处理物业业务数据时，不暴露不必要的工单流转入口

## 9. 统计与报表设计

统计优先返回前端直接可消费的聚合结构，避免前端自行拼装。

输出形式：

- 总览卡片值
- 按月份趋势数据
- 分类分布数据
- 待办清单数据

统计时间范围先支持：

- 最近 7 天
- 最近 30 天
- 当前年度按月

## 10. 错误处理与兼容策略

- 保留现有统一返回结构 `Result`
- 前端接口层继续使用现有 `axios` 封装
- 对新增字段采用兼容写法，避免旧页面在改造过程中完全不可用
- 文件上传失败、越权访问、状态流转非法时，返回明确错误信息

## 11. 测试策略

后端：

- 以服务层和控制器关键业务流程为主补测试
- 覆盖登录、角色权限、房产绑定、账单生成、报修状态流转、公告发布、系统配置维护、统计接口

前端：

- 当前仓库未见现成测试体系，优先保证构建通过与手工回归
- 对关键页面进行联调验证：登录、角色跳转、列表查询、表单提交、状态变更

## 12. 实施顺序

建议实现顺序：

1. 后端数据模型与 SQL 增补
2. 用户与权限接口修正
3. 房产、账单、报修、公告四大核心域重构
4. 系统配置与操作日志
5. 统计接口
6. 前端角色化布局与路由调整
7. 各业务页面改造
8. 联调与回归

## 13. 验收标准

- 三种角色均可独立登录并看到正确菜单
- 业主端核心闭环可完整使用
- 物业管理员端可完成日常运营管理
- 系统管理员端可完成管理员管理与系统配置
- 统计页面能展示真实后端数据
- 关键操作具备权限校验和操作日志
- 保留现有基础框架代码，不进行无关替换
